Socket Communication

Handshake

The WebSocket protocol starts with an initial handshake between the client and the server, which usually occurs over a standard HTTP connection. During this handshake, the client sends an HTTP request with a special Upgrade header indicating its desire to establish a WebSocket connection. If the server supports WebSocket and approves the handshake, it responds with an HTTP 101 status code (Switching Protocols), indicating a successful upgrade to the WebSocket protocol. From this point onwards, the connection transitions to full-duplex mode, allowing both the client and server to send messages asynchronously.

Data Transfer

After the handshake, the WebSocket connection is established, and both the client and server can send messages to each other at any time without waiting for a request. Messages can be sent as text or binary data, and WebSocket frames encapsulate these messages. The protocol supports various types of messages, including text frames, binary frames, ping frames (used for keep-alive checks), and pong frames (responses to ping frames).

Closing the Connection

Either the client or server can initiate the closing of the WebSocket connection by sending a special close frame. Upon receiving the close frame, the receiving party acknowledges the request and sends its own close frame, after which the connection is considered closed.

Advantages of WebSocket Communication

  • Low latency: WebSocket communication eliminates the overhead of repeatedly establishing and tearing down connections for each request-response cycle, resulting in lower latency.
  • Efficient: WebSocket communication uses a single, long-lived connection for bidirectional communication, reducing the overhead of multiple HTTP connections.
  • Full-duplex communication: Unlike HTTP, which follows a request-response pattern, WebSocket allows for simultaneous data transfer in both directions, enabling more interactive and real-time applications.
